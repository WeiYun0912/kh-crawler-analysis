§ 01 · Problem

為什麼需要這套規則？

爬蟲在判斷「同一份內容是否已存在」時，使用 md5(markdown + title + meta_data + hashVersion) 比對。 但某些網站的頁面內嵌「列表頁返回連結」、「分頁器」等元素，URL 中含有 page=N 等位置相關參數，會隨著該文章在來源列表中的位置漂移而變動。

雖然主體內容完全相同，但 markdown 文字差幾個字元，hash 就完全不同， 造成系統把舊資料「刪除 + 新增」當成異動處理。

實測案例：coia 站點

對比第一次與第二次爬到的 coia 全站資料（各約 1,054 個 JSON）：

項目 數量
被視為「新增/刪除」的檔案A 獨有 163 + B 獨有 163
其中真正內容異動僅 6 + 6 ≈ 12 個
假異動率約 96%

主要原因是 detail 頁底部的「回到列表」連結：

html 
<a href="cms_emphasis.php?...&page=1" title="回到衛生福利組列表">返回</a>
<a href="cms_emphasis.php?...&page=2" title="回到衛生福利組列表">返回</a>

1,118 字的 markdown 中只差 1 個字元（page=1 vs page=2），但 md5 hash 完全不同。

§ 02 · Solution

Hash 計算與內容入庫
分離為兩條路徑

路徑 函數 結果用途
入庫用 filterAndConvertHtml(html, baseUrl) 寫入 crawler_documents_content.content、向量化、搜尋 — 保留完整 a href
Hash 用 filterAndConvertHtmlForHash(html, baseUrl, siteId) 僅供 md5 計算 — 套用站點專屬清洗規則

兩條路徑算出來的 markdown 不同：入庫的保留 URL，hash 的可以洗掉 URL 雜訊。對搜尋、向量、RAG 無任何影響。

§ 04 · Common rules

目前支援的共用規則

定義在 utils/crawler/rules/common-rules.js。

stripAHrefQueryParams Available

從所有 <a> 的 href 移除指定 query 參數。

Options
{ params: string[] }
適用情境
只想清掉特定 query 參數，保留其他 URL 結構
效果
<a href="x.com/?id=5&page=1"> → <a href="x.com/?id=5">
js · 範例設定 
{ rule: "stripAHrefQueryParams", options: { params: ["page", "timestamp"] } }
stripAllAHref In Use · coia

無差別移除所有 <a> 的 href 屬性。連結文字保留，turndown 後變純文字。

Options
無
適用情境
該站點的所有內嵌 URL 都可能是雜訊，最直接的方式是讓 hash 無視所有 a href
效果
<a href="x.com/?id=5">點我</a> → turndown 後變 點我
js · 範例設定 
{ rule: "stripAllAHref" }
§ 05 · Configuration

怎麼幫一個站點啟用規則

編輯 utils/crawler/rules/site-rules.js，在 SITE_RULES 加一筆：

js · site-rules.js 
const SITE_RULES = {
    // key 是 crawler.site_id 欄位（不是 UUID）
    coia: [
        { rule: "stripAllAHref" },
    ],

    // 可以同時啟用多條規則
    "其他-site-id": [
        { rule: "stripAHrefQueryParams", options: { params: ["page", "timestamp"] } },
        { rule: "stripAllAHref" },
    ],
};

執行下一次同步，log 開頭會看到摘要：

stdout · log 
[hash-rules] siteId=coia：啟用 1 條規則 → stripAllAHref
[hash-rules] siteId=其他: 未啟用站點規則，hash 使用預設清洗邏輯
§ 06 · Extension

怎麼新增第三條規則

在 utils/crawler/rules/common-rules.js 的 COMMON_RULES 加一條：

js · common-rules.js 
const COMMON_RULES = {
    // ... 既有規則 ...

    // 範例：移除所有 iframe（包含 google maps 等嵌入）
    removeAllIframes: {
        description: "移除所有 iframe",
        apply: ($) => { $("iframe").remove(); },
    },
};

加完後立刻可在 site-rules.js 用 { rule: "removeAllIframes" } 啟用，不用改其他檔案。

規則 apply 函數的簽名

ts · signature 
apply: (cheerio$, options) => void
  • cheerio$ — cheerio 已載入的 DOM，直接操作
  • options — 來自 site-rules.js 的 options（可選）
  • 不需要 return；直接修改 cheerio$ 即可
  • 不要 throw — 失敗時內部處理；若真的 throw，外層有 try/catch 接住，不會中斷流程
§ 07 · Resilience

安全性與錯誤處理

系統設計為「規則出錯絕不影響整體流程」，分三層保護：

i

單條規則出錯 — 跳過該規則、繼續算 hash

applySiteHashRules 對每條規則的 apply() 都包了 try/catch：規則 throw 時 console.error + 跳過，未知規則名稱則 console.warn + 跳過。多條規則中只有一條壞掉，其他規則照常套用。

ii

filterAndConvertHtmlForHash 整個函數出錯 — fallback 回原邏輯

若 cheerio / turndown 在新路徑整個失敗，會 fallback 使用 filterAndConvertHtml。此情況下：該檔案 hash 不會套用站點規則（等同沒裝這套規則），但 hash 仍會被算出來，檔案處理不中斷。

iii

processFile 整個出錯 — 該檔案跳過、不影響其他檔案

這是 existing behavior，不是新引入的。processCrawlerFiles.js 的 queue worker 對每個 processFile 都有 try/catch，單一檔案出問題會被 log 記錄並跳過，不影響整個同步流程。

§ 08 · Side effect

首次啟用會有一次性陣痛

⚠ Warning

任何時候對某站點啟用 / 修改 / 移除規則，該站點下一次同步時，所有檔案會被視為「舊全刪 + 新全增」一次。

原因：規則改變後，同一份 html 算出的 markdown 變了，hash 自然不同 → 對不上 DB 既有 hash → 系統判定為「舊資料消失 + 新資料新增」。之後就會回到正常水位。

建議：規則設定盡量一次到位、別頻繁變動，每次變動都會付一次「全站重洗」的代價。

§ 09 · Performance

性能影響

站點配置 額外開銷
沒設規則 零 — hasSiteHashRules(siteId) 回 false 後直接走原本路徑
有設規則 每筆檔案多跑一次 cheerio.load + turndown

對「有設規則的站點」會增加 ~50% 的 HTML 處理時間。在 coia 等小站（< 5,000 筆）感受不到，在大站建議實測再評估。
