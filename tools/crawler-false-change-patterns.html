§ 02 · Diagnostic workflow

排查流程

  1. 拿到「最近兩次爬取」的 JSON 資料夾，丟到比對腳本的 input 資料夾
  2. 跑 compare-by-hash.js 確認假異動數量與真實 URL 變動的比例
  3. 跑 analyze-48-diff.js 取得差異模式分類
  4. 對照下方類型表，找出兇手
§ 03 · Tooling

排查工具

放在 D:/sandbox/crawler-compare-json/，可以複製或修改後重用。

腳本 用途
compare.js 兩個資料夾的「URL / 檔名層級」比對，產出 Excel
compare-by-hash.js 用 production hash 邏輯（含站點規則）比對，找出真正會被當成新增/刪除的對
analyze-48-diff.js 對「URL 相同但 hash 不同」的對做差異點分析、自動分類
list-all-50.js 列出全部假異動配對的檔名清單
§ 04 · The eight types

八種類型詳述

01

URL page 參數有/無

Solved
特徵
兩邊 URL 只差 ?page=N 或 &page=N
根本原因
爬蟲對「同一頁的不同入口」當成不同 URL；同一頁可能從列表第 1 頁（含 &page=1）或從詳細頁（無 page）連過去
解法
stripAllAHref 規則一併清掉
歷史佔比
早期占假異動的 88%（144 / 163）
檔名範例
cms_subsidy_detail_php_id_2174_..._Services2025072500004.html.json cms_subsidy_detail_php_id_2174_..._Services2025072500004_page_1.html.json
02

內嵌「回列表」連結的 page=N 漂移

Solved
特徵
detail 頁底部「回到 XXX 列表」連結的 page 參數變了。markdown 1118 字裡只差 1 個字元
根本原因
detail 頁的「回列表」連結帶有「該文章在列表中的頁碼」，每當新公告插入列表，舊文章的頁碼就會變
解法
stripAllAHref 規則
歷史佔比
早期占假異動 79%（129 / 163）的主因
差異範例
A: ...&n=Emphasis&page=1 "回到衛生福利組列表" B: ...&n=Emphasis&page=2 "回到衛生福利組列表"
03

URL 重複 query 參數（爬蟲 bug）

Solved
特徵
URL 包含 appId=X&appId=X 這種重複 key
根本原因
爬蟲組 URL 時把同個 query key 加了兩次（程式 bug）。也可能不同次爬取產出 appId=X vs appId=X&appId=X 兩種 URL
解法
stripAllAHref 規則（連同其他 URL 雜訊一起被清）
附註
這是「爬蟲程式 bug」，根本上應該修爬蟲組 URL 的邏輯，但目前用 hash 規則回避也有效
檔名範例
news_php_n_news_appId_Job_appId_Job_page_1.html.json
04

URL 額外參數漂移（appId2 等）

Solved
特徵
兩邊 URL 只差一個語意無關的 query 參數
解法
stripAllAHref 規則
檔名範例
data_php_n_Download_appId_BusinessReport.html.json data_php_n_Download_appId_BusinessReport_appId2_1_page_1.html.json
05

詳細頁的「瀏覽次數」漂移

Solved
特徵
detail 頁包含一個獨立的點閱數 <td>，隨用戶瀏覽自然增加。差異片段是純數字（1–5 字）
根本原因
頁面 HTML 結構是 <th>瀏覽</th> 與顯示數字的 <td> 分屬不同 cell，中間沒有「瀏覽:123」這種 inline 結構，utils/crawler.js 的後處理 regex 抓不到
解法
clearTableColumnsByHeader 規則 — 偵測 thead 含「瀏覽/點閱/下載」並清空對應 column
歷史佔比
引入新規則前佔假異動的 16%（8 / 50）
檔名範例
cms_subsidy_php_appId_Services2014092290853_n_Services_page_1 點閱: 1 → 5 cms_subsidy_php_appId_Services2021020120131_n_Services_page_1 點閱: 46 → 51 data_php_n_Download_appId_Avoid 點閱: 0 → 2
06

列表頁多筆公告的「瀏覽次數」漂移

Solved
特徵
  • 差異片段大（數十到數百字）
  • 看起來像中文公告標題在差，但仔細對照其實公告標題完全相同，只有夾在標題之間的數字變了
根本原因
與類型 05 機制相同，但因為列表頁有多筆 → 每筆的點閱數都會漂移 → markdown diff 範圍很大、看起來像「整個列表都變了」
解法
clearTableColumnsByHeader 規則
歷史佔比
引入新規則前佔假異動的 84%（42 / 50） — 規則上線後 0
檔名範例
cms_bulletin_php_appId_hrBulletin_n_about 差異 423 字（多筆 +1~5） data_php_n_Download_appId_Statistics_appId2_1_page_1 差異 338 字 meeting_php_n_about_appId_Meeting 差異 194 字
規則設定
js · 在 site-rules.js 啟用 
{ rule: "clearTableColumnsByHeader" }
// 用預設關鍵字：瀏覽 / 點閱 / 下載 / 次數 / 觀看 / 查看 / 閱讀
運作方式：對每個 <table> 檢查 <thead>，找出 column header 含關鍵字的 index，把 <tbody> 對應 column 的 <td> 內容清空
07

日期 / 時間戳漂移

Observing
特徵
頁面顯示「最後更新時間」、「爬取時間」、「現在時間」之類的時間戳，每次爬取都不同
狀態
尚未在 coia 站實際遇到，但邏輯上可能發生
預期解法
類似類型 05/06 的 regex 擴充，或在 cheerio 階段移除含特定 class 的元素
08

動態 DOM id 漂移

Historical
特徵
DOM 元素的 id 屬性每次載入頁面都不同，例如 id="Group_5_4620963338"
狀態
歷史問題，hash v2.3 已修
修法
將 hash 計算從 html 改為 markdown — turndown 不會把 id 屬性帶到 markdown 輸出
09

Footer「您是第 N 位瀏覽者」訪客計數器

Solved
特徵
頁面 footer 顯示「您是第 1976474 位瀏覽者」整站共用，每次爬都不同。HTML 通常是 您是第<strong>1976474</strong>位瀏覽者
根本原因
訪客計數器是 server-side 動態渲染，整站每頁都有 → 瞬間製造數千個假異動
解法
filterAndConvertHtml 的 inline regex（容忍中間有 HTML tag）
歷史影響
busker 站實測，每天約 593 對假異動同時來自類型 9 + 10
檔案範例
busker_kcg_gov_tw_artist_Details_Parser_99_6_27____1076.html.json (共影響 ~593 對檔案)
10

Footer「更新日期 YYYY-MM-DD」site-wide 時間戳

Solved
特徵
頁面 footer 顯示「更新日期 2026-05-25」整站共用且每天會變。HTML 通常是 更新日期<strong>2026-05-25</strong>
根本原因
跟類型 9 類似，是 server-side 動態渲染當下日期。注意：跟文章「真實的最後更新日期」（通常在 <time> 或 metadata box）不同
解法
filterAndConvertHtml 的 inline regex，匹配 更新日期/最後更新/最近更新/今日日期/查詢日期/系統日期/目前時間 + 日期
副作用提醒
如果某站點文章 metadata 用「更新日期：YYYY-MM-DD」這種純文字格式（很少見，通常用 <time>），會被誤殺。建議啟用後跑 compare 確認
檔案範例
busker_kcg_gov_tw_artist_Details_Parser_99_6_27____1076.html.json
§ 05 · Quick judgment

真實 vs 假異動 速判表

觀察 真實異動 假異動
差異片段是純數字（1–5 字） 罕見 很可能是點閱數
差異片段散落 markdown 多處、看起來像列表 罕見 列表頁點閱數漂移
差異片段是新的公告標題 / 段落 是 不會
差異片段是日期 (YYYY-MM-DD) 可能（新公告） 也可能（時間戳）
URL 完全不同 通常是 也可能（URL 噪音）
同一站點短時間內 > 10 筆新增/刪除 罕見 很可能
§ 06 · Standard procedure

看到「大量新增/刪除」時的 SOP

  1. 先看數量 — 一個站點短時間內新增/刪除 > 10 筆，第一直覺先當假異動處理。不要急著回報「網站新增/下架了 50 個頁面」— 先驗證。
  2. 收集兩次同步的 JSON — 從爬蟲 output 撈出最近兩次的 JSON（或從 DB 反推），分別放到比對腳本的 input 資料夾。
  3. 跑分析腳本 — node compare-by-hash.js（用 production hash 邏輯比對），node analyze-48-diff.js（分析差異模式）。
  4. 對照類型總表 — 找出屬於哪一類。
  5. 依照狀態行動：類型 1–4 確認 stripAllAHref 已啟用；類型 5–6 等待新規則或先排除列表頁；新模式則加進本文件。
⚠ 規則上線後的副作用

每次新增 / 修改 / 移除規則，該站點下次同步會一次性把所有檔案視為「新刪 + 新增」（因為新舊 hash 不同）。屬於可預期的一次性陣痛，不要混淆成新問題。詳見 crawler-site-hash-rules.html 的「站點啟用規則的副作用」一節。
